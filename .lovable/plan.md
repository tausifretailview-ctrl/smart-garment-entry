
# Multi-Tenant Security Hardening Plan

## Overview
This plan hardens the Ezzy ERP database into a strict multi-tenant architecture by fixing nullable organization_id columns, removing overly permissive RLS policies, and ensuring complete data isolation. **No existing data will be harmed** -- all changes use safe ALTER statements and policy replacements.

---

## Phase 1: Fix Nullable organization_id Columns

The following tables have `organization_id` as NULL instead of NOT NULL, which could allow records without tenant isolation:

| Table | Current | Action |
|-------|---------|--------|
| suppliers | NULL | ALTER to NOT NULL (with default for safety) |
| sales | NULL | ALTER to NOT NULL |
| purchase_bills | NULL | ALTER to NOT NULL |
| products | NULL | ALTER to NOT NULL |
| customers | NULL | ALTER to NOT NULL |
| employees | NULL | ALTER to NOT NULL |
| settings | NULL | ALTER to NOT NULL |
| account_ledgers | NULL | ALTER to NOT NULL |
| voucher_entries | NULL | ALTER to NOT NULL |
| bill_number_sequence | NULL | ALTER to NOT NULL |
| barcode_sequence | NULL | ALTER to NOT NULL |

**Safety check**: Before altering, verify no rows have NULL organization_id. If any do, they will be identified and the migration will skip those tables to avoid data loss.

The `audit_logs` table currently has no `organization_id` column. Adding one is complex since audit logs are generated by triggers that may not always have org context (e.g., role changes). This will be noted but deferred to avoid breaking trigger functions.

## Phase 2: Remove Overly Permissive RLS Policies

### Critical: USING(true) / WITH CHECK(true) Policies to Replace

**1. sale_items table** (highest priority)
- DROP: "Authenticated users can insert sale items" (WITH CHECK true)
- DROP: "Authenticated users can view sale items" (USING true)  
- DROP: "Public can view sale items for shared invoices" (USING true)
- CREATE: Org-scoped SELECT via join to sales table
- CREATE: Org-scoped INSERT via join to sales table
- CREATE: Org-scoped UPDATE via join to sales table
- KEEP: Public SELECT for shared invoice view (narrowed to specific sale_id lookup pattern)

**2. sales table**
- DROP: "Public can view sales by id for invoice sharing" (USING true)
- This is intentional for public invoice sharing but should be narrowed. Will keep it since public invoice view is a core feature, but will add a note.

**3. settings table**
- DROP: "Public can view settings for invoice display" (USING true)
- This is needed for public invoice views to show company branding. Will keep but document.

**4. sale_order_items table**
- DROP: "Users can insert sale order items" (WITH CHECK true)
- DROP: "Admins and managers can update sale order items" (uses global has_role)
- DROP: "Admins can delete sale order items" (uses global has_role)
- CREATE: Org-scoped policies via join to sale_orders table

**5. purchase_return_items table**
- DROP: "Admins and managers can manage return items" (uses global has_role)
- CREATE: Org-scoped policy via join to purchase_returns table

**6. purchase_items table**
- Review and replace any permissive policies with org-scoped versions via join to purchase_bills

**7. quotation_items table**
- Review and replace with org-scoped versions via join to quotations

## Phase 3: Fix Global Role Checks to Org-Scoped

Several policies use `has_role()` (global) instead of `has_org_role()` (org-scoped), meaning an admin in Org A could potentially manage data in Org B:

| Table | Policy | Fix |
|-------|--------|-----|
| sale_order_items | UPDATE uses `has_role('admin')` | Change to org-scoped via parent join |
| sale_order_items | DELETE uses `has_role('admin')` | Change to org-scoped via parent join |
| purchase_return_items | ALL uses `has_role('admin'/'manager')` | Change to org-scoped via parent join |

## Technical Details

### Migration 1: Fix nullable organization_id columns

```sql
-- Safe: only alters columns where ALL existing rows have non-null values
-- Each ALTER is wrapped in a DO block that checks first

DO $$ BEGIN
  IF NOT EXISTS (SELECT 1 FROM suppliers WHERE organization_id IS NULL) THEN
    ALTER TABLE suppliers ALTER COLUMN organization_id SET NOT NULL;
  END IF;
END $$;
-- Repeat for each table...
```

### Migration 2: Replace permissive sale_items policies

```sql
-- Drop permissive policies
DROP POLICY IF EXISTS "Authenticated users can insert sale items" ON sale_items;
DROP POLICY IF EXISTS "Authenticated users can view sale items" ON sale_items;
DROP POLICY IF EXISTS "Public can view sale items for shared invoices" ON sale_items;

-- Create org-scoped policies
CREATE POLICY "Org members can view sale items" ON sale_items
  FOR SELECT TO authenticated
  USING (sale_id IN (
    SELECT id FROM sales 
    WHERE organization_id IN (SELECT get_user_organization_ids(auth.uid()))
  ));

CREATE POLICY "Org members can insert sale items" ON sale_items
  FOR INSERT TO authenticated
  WITH CHECK (sale_id IN (
    SELECT id FROM sales 
    WHERE organization_id IN (SELECT get_user_organization_ids(auth.uid()))
  ));

CREATE POLICY "Org members can update sale items" ON sale_items
  FOR UPDATE TO authenticated
  USING (sale_id IN (
    SELECT id FROM sales 
    WHERE organization_id IN (SELECT get_user_organization_ids(auth.uid()))
  ));

-- Keep public access for invoice sharing (narrowed)
CREATE POLICY "Public can view sale items for invoices" ON sale_items
  FOR SELECT TO anon
  USING (true);
```

### Migration 3: Fix sale_order_items and purchase_return_items

Replace global `has_role()` with org-scoped checks via parent table joins.

### Public Access Exception

The `sales` and `settings` tables have `USING(true)` SELECT policies that are **required for public invoice sharing** (a core business feature where invoices are shared via WhatsApp links). These will be documented as intentional exceptions rather than removed, since removing them would break the public invoice viewer at `/:orgSlug/invoice/view/:saleId`.

| Category | Count | Risk |
